# aocl

Rust bindings for the [AMD Optimizing CPU Libraries (AOCL)](https://www.amd.com/en/developer/aocl.html).

This workspace contains two crates:

- [`aocl-sys`](crates/aocl-sys/) — Raw `unsafe extern "C"` FFI bindings, generated by `bindgen` against the AOCL C headers. One Cargo feature per AOCL component.
- [`aocl`](crates/aocl/) — Safe, idiomatic Rust wrappers built on top of `aocl-sys`. Typed errors, RAII handles, slice-based APIs.

## Status

Early development. The workspace is scaffolded with feature flags for each AOCL component; bindings and safe wrappers are added per-component. See each crate's README for the matrix of what is wired up today.

## Supported AOCL components

The crates expose Cargo features for the following AOCL components (mirrored across both crates):

- `blis`              — AOCL-BLAS (BLIS): BLAS Level 1/2/3
- `libflame`          — AOCL-LAPACK (libFLAME): LAPACK / LAPACKE
- `fftw`              — AOCL-FFTW: FFTW-compatible FFT
- `libm`              — AOCL-LibM: vectorized math functions
- `sparse`            — AOCL-Sparse: sparse BLAS / solvers
- `rng`               — AOCL-RNG: random number generation
- `securerng`         — AOCL-SecureRNG: hardware-backed RNG
- `utils`             — AOCL-Utils: CPU identification, threading helpers
- `compression`       — AOCL-Compression: LZ4/ZLIB/ZSTD/LZMA/BZIP2/Snappy
- `crypto`            — AOCL-Cryptography (ALCP)
- `data-analytics`    — AOCL-DA: data analytics / ML primitives
- `scalapack`         — AOCL-ScaLAPACK: distributed LAPACK (requires MPI)

Plus build-mode features:

- `ilp64`        — use 64-bit integer indexing (default is LP64 / 32-bit)
- `static-link`  — link AOCL libraries statically (default is dynamic)

## Prerequisites

1. **Install AOCL** from <https://www.amd.com/en/developer/aocl.html>.
   - Windows default: `C:\Program Files\AMD\AOCL-Windows`
   - Linux default:   `/opt/AMD/aocl/aocl-linux-*`
2. **Set `AOCL_ROOT`** to your install root if it is not in the default location.
3. **Install `libclang`** (required by `bindgen`):
   - Windows: install LLVM and set `LIBCLANG_PATH`, or install via `winget install LLVM.LLVM`.
   - Linux:   `apt install libclang-dev` (or distro equivalent).

## Build

```sh
# Default features (utils only — minimal cost to compile)
cargo build

# Just the BLAS bindings, dynamically linked, LP64
cargo build -p aocl --no-default-features --features blis

# Everything, ILP64, static linking
cargo build -p aocl --no-default-features \
  --features "blis libflame fftw libm sparse rng utils compression crypto data-analytics ilp64 static-link"
```

## Runtime

When linking dynamically (the default), the AOCL DLLs / shared objects must be reachable at runtime:

- **Windows**: add the relevant `<AOCL_ROOT>\<component>\lib[\LP64|\ILP64]` directory to `PATH` before running. Some components live under `lib\<arch>\shared\` (sparse, fftw, compression, data-analytics). For example:

  ```bat
  set PATH=C:\Program Files\AMD\AOCL-Windows\amd-utils\lib;%PATH%
  set PATH=C:\Program Files\AMD\AOCL-Windows\amd-blis\lib\LP64;%PATH%
  ```

  Many of AMD's Windows AOCL DLLs are built with the Intel compiler and depend on `libmmd.dll` and `svml_dispmd.dll` (Intel runtime). If you have Intel oneAPI installed, add `C:\Program Files (x86)\Intel\oneAPI\<version>\bin` to `PATH` as well. Some components (`compression`) also pull in LLVM's OpenMP runtime (`libomp.dll`) — add `C:\Program Files\LLVM\bin` to `PATH` if you have LLVM installed. The `crypto` ALCP DLL also depends on OpenSSL 3 (`libcrypto-3-x64.dll`).

  When mixing components from different AOCL DLLs that link different OpenMP runtimes (Intel `libiomp5md.dll` vs LLVM `libomp.dll`), set `KMP_DUPLICATE_LIB_OK=TRUE` to suppress the runtime's duplicate-library abort. AMD's libraries do not actually conflict in practice.

- **Linux**: add to `LD_LIBRARY_PATH`, or rely on the system loader cache.

For builds with `--features static-link` no runtime configuration is required.
